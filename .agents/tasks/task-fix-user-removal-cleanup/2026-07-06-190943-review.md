# Complete agent cleanup on EID removal

The DELETE `/api/admin/eids/:eid` endpoint previously only deleted the EID from `allowedEids`, leaving agent runtime records, number assignments, followup locks, and interested-lead ownership scattered across application state. This change adds cascading cleanup: it deletes the agent record, nullifies all number-field references, emits a socket event to force-disconnect the agent, broadcasts updated admin stats, and filters removed agents from the rankings endpoint.

**Watch for:** The `dialedBy` field on numbers is not cleared (confirmed), so a removed agent's ID still blocks those numbers from being re-served to other agents. The `force-stop-agent` event is broadcast to all connected sockets (confirmed) but no handler for it exists on the server side or is shown in this diff — its effectiveness depends entirely on client-side code not visible here.

## High-level view

The deletion handler now performs five cleanup steps in sequence: remove the EID, remove the agent runtime record, sweep all numbers for references, persist state, then emit socket events. The sweep covers `assignedTo`, `followupLockedBy`, and `interestedBy` — but misses `dialedBy`, which is the field `getNextNumber` uses to determine whether a number has already been served. Numbers previously dialed by the removed agent remain marked as dialed and won't re-enter circulation until the next daily reset.

The rankings endpoint now filters out agents whose EID no longer exists in either `allowedEids` or `appState.agents`, preventing ghost entries in the leaderboard. Historical `dialedLog` entries for the removed agent persist and are trimmed by the existing 90-day TTL during daily reset.

The `force-stop-agent` socket event is a broadcast (`io.emit`) rather than a targeted emit to the specific socket. Every connected client — including admin sockets — receives the event for every removal.

## The `dialedBy` gap in number cleanup

The cleanup loop iterates `appState.numbers` and nullifies three fields: `assignedTo`, `followupLockedBy`, and `interestedBy`. The `dialedBy` field is absent. In `getNextNumber`, the very first filter condition is:

```javascript
if (n.dialedBy || n.assignedTo) return false;
```

A number with `dialedBy` set to the removed agent's ID will never be returned by `getNextNumber` for any agent. The daily reset (line ~225) nullifies `dialedBy` for non-followup/non-interested numbers, so they recover the next day — but until then they're locked out of circulation.

If the removed agent was mid-shift and had dialed 50 numbers today, those 50 numbers are inaccessible to all other agents until tomorrow. The fix: add `if (n.dialedBy === agentId) { n.dialedBy = null; n.dialedAt = null; }` to the existing cleanup loop.

## Socket disconnect lifecycle

After EID removal, `appState.agents[agentId]` no longer exists, so the socket `disconnect` handler's `if (agent)` guard prevents a crash — but the socket remains connected until the client acts on `force-stop-agent` or the TCP connection drops.

The existing pattern for forced disconnection is `io.emit('force-stop')` (used by three admin endpoints), which tells all agent clients to stop. The new `force-stop-agent` event is a targeted variant. However, there's no evidence in this diff that the client handles it. If it doesn't, the removed agent can still trigger `agent-next`, and the server will respond with `'Agent not found'` errors since the agent record is deleted. The fail mode is noisy errors rather than data corruption — fail-closed.
